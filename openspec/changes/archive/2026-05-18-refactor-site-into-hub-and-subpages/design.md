## Context

当前项目是一个静态网页仓库，业务内容集中在根目录 `index.html` 中。首页直接承载“你画我猜”页面结构、按钮交互、二维码展示和词库脚本引用；词库数据位于 `config.js`，以全局变量方式注入页面。站点目前没有主站入口层、子页面目录规则、公共页面骨架或统一的扩展方式。

这次变更属于跨页面和跨资源组织方式的重构，不只是样式调整。它需要同时调整站点信息架构、页面入口关系、静态资源划分方式，以及现有“你画我猜”页面的落位方式，因此需要在编码前先明确目录、路由和资源边界。

约束如下：
- 项目是纯静态站点，目标结构应兼容 GitHub Pages 一类目录即路由的部署方式。
- 当前仓库使用较老的前端资源目录，如 `vendor/`、`dist/` 和内联脚本；重构需要尽量降低耦合，但不要求一次性移除所有历史依赖。
- 现有“你画我猜”核心行为需要保留，包括词条随机展示和单次选择锁定。

## Goals / Non-Goals

**Goals:**
- 将站点重构为主站首页加多个独立子页面的结构。
- 将“你画我猜”迁移到独立子页面，并提供稳定独立链接。
- 让主站首页只负责聚合和导航，以卡片形式按固定顺序展示子页面入口。
- 建立可复用的页面目录规范、公共资源目录和新增页面约定。
- 将“你画我猜”的页面逻辑与词库数据从首页解耦，避免继续依赖全局脚本堆叠。

**Non-Goals:**
- 不在本次设计中引入构建工具、前端框架或后端服务。
- 不要求本次重构同时完成所有旧依赖的彻底清理和替换。
- 不扩展“你画我猜”玩法规则，只做页面承载位置与资源组织的重构。
- 不在本次变更中新增大量业务页面；重点是建立后续可扩展的结构。

## Decisions

### 1. 使用目录路由作为页面组织方式

采用 `index.html` 作为主站入口，并为每个子页面创建独立目录，例如 `pages/draw-and-guess/index.html`。

原因：
- 与静态托管平台天然兼容，不需要额外路由器或重写规则。
- 每个子页面都能自然获得独立链接，例如 `/pages/draw-and-guess/`。
- 新增页面时只需要复制目录模板，心智成本低。

备选方案：
- 继续使用单页站点，通过锚点或条件渲染切换内容。否决原因是页面职责会继续混杂，不满足独立 URL 的需求。
- 使用单文件多页面，如 `draw-and-guess.html` 直接放根目录。可行但扩展时命名易散乱，不如目录化结构稳定。

### 2. 主站首页采用数据驱动的卡片列表

主站首页不再手写每一个子页面入口，而是通过一个统一的数据文件维护页面卡片顺序、标题、简介和链接，再由首页渲染卡片列表。

原因：
- 页面列表顺序是明确需求，数据驱动更容易维护和扩展。
- 新增页面时只需新增目录和一条配置，避免多处同步修改 HTML。
- 可降低首页与子页内容之间的耦合。

备选方案：
- 直接在首页硬编码卡片 HTML。短期成本更低，但后续增加子页面时重复改动较多，容易漏改排序和链接。

### 3. 将“你画我猜”拆分为独立页面逻辑与独立数据源

“你画我猜”页面的 DOM、交互逻辑和词库数据从根首页剥离。页面脚本放入专属资源文件，词库放入独立数据文件，由子页面自行加载。

原因：
- 当前 `config.js` 的全局变量方式不适合多页面演进。
- 独立页面需要拥有自洽资源边界，避免首页残留业务脚本。
- 后续如果增加新的词条分类或新的小游戏，可以沿用相同模式。

备选方案：
- 保留 `config.js` 作为全站共享全局词库。否决原因是会让非游戏页面仍然承受无关脚本和数据，扩展性差。
- 直接将词库硬编码进子页面 HTML。否决原因是数据维护不集中，不利于后续扩展分类。

### 4. 公共样式与页面专属样式分层

建立 `assets/css/` 和 `assets/js/` 作为公共资源目录。主站和所有子页面共享基础样式与页面骨架；页面专属逻辑和样式放到各自命名文件中。

原因：
- 主站和子页面会共享头部、页脚、返回主站入口、基础排版等通用结构。
- 样式分层可以降低未来新增页面时的相互污染风险。
- 允许项目在保留部分历史依赖的同时，逐步把实际业务样式迁移到清晰的新目录中。

备选方案：
- 继续沿用 `dist/` 与 `vendor/` 中的所有资源并在每个页面直接引用。否决原因是依赖边界不清，无法体现新的站点结构。

### 5. 采用渐进式迁移，而非一次性重写所有资源

先完成信息架构重组、页面拆分和资源落位，再视情况逐步裁剪旧的主题依赖。

原因：
- 当前站点体量小，但历史依赖较旧，完全一次性替换的收益不一定高。
- 渐进式迁移更适合控制风险，先保证独立 URL、主站卡片聚合和页面可扩展性达标。

备选方案：
- 在本次变更里彻底去除所有旧依赖并重写样式。可做，但会显著扩大范围，冲淡本次重构的核心目标。

## Risks / Trade-offs

- [旧资源残留导致新旧结构混用] → 在迁移时明确首页只保留主站职责，业务逻辑全部移出根页面，并在设计中定义新的资源入口目录。
- [卡片数据与真实页面目录不一致] → 约定卡片配置必须包含稳定 slug，链接由 slug 推导或统一填写，避免首页写死散乱路径。
- [相对路径在子页面中容易写错] → 统一规定子页面引用公共资源时使用清晰的相对层级，并在实现阶段优先验证首页与 `draw-and-guess` 页面。
- [逐步迁移会短期保留旧依赖] → 接受过渡期成本，但将新增页面全部建立在新目录和新资源约定之上，避免继续扩大历史包袱。
- [后续页面样式需求变多后公共样式膨胀] → 将页面专属样式限制在独立文件中，公共层只保留骨架、排版和共通组件。

## Migration Plan

1. 建立新的主站与子页面目录结构，以及公共资源目录。
2. 将根目录 `index.html` 重构为主站首页，只保留站点介绍、卡片列表和公共导航。
3. 新建 `pages/draw-and-guess/index.html`，迁移现有“你画我猜”页面结构与交互。
4. 将词库从 `config.js` 拆分到独立数据文件，并调整“你画我猜”子页面的加载方式。
5. 引入首页卡片配置数据，使主站按顺序展示各子页面入口。
6. 验证首页与子页面链接、返回主站入口、资源路径和核心交互是否正常。
7. 视实现情况决定是否移除未再使用的旧引用或保留为过渡依赖。

回滚策略：
- 如果迁移后页面异常，可临时恢复根目录首页直接承载原游戏内容的旧入口。
- 因为项目是静态站点，回滚本质上是恢复旧文件结构和旧引用关系，不涉及数据迁移。

## Open Questions

- 主站首页的卡片数据使用 JSON 还是 JS 配置文件承载，需在实现时根据浏览器兼容性和维护便利性最终确定。
- 是否保留现有 Bootstrap 风格作为过渡视觉基础，还是在实现时同步收敛为更轻量的自定义样式。
- 是否需要同时提供一个示例空白子页面模板，作为后续新增页面的复制起点。
