## Context

当前项目已经采用“主站首页 + 独立子页面”的静态站结构，首页卡片由 `assets/js/site-data.js` 驱动，已有子页面通过 `pages/<slug>/index.html` 和公共样式 `assets/css/site.css` 统一承载。但 `COAController.html` 仍然保留在仓库根目录，使用独立的一套页面结构、内联样式与内联脚本，没有纳入现有子页面目录和作品集首页入口。

该页面本身已经具备完整业务能力，包括：
- 读取指定 Gist 中的 `status.json`
- 向 `command.json` 发送固定动作
- 本地缓存 GitHub Token 和账号别名
- 通过页面内冷却与状态提示限制频繁操作

当前最大的不一致点有两个：
- 页面信息架构不一致：未通过主站首页进入，也未复用统一页面骨架和返回首页导航。
- 配置模型不一致：`GIST_ID` 被写死在脚本常量中，而 Token 采用本地输入缓存方式，二者对“连接目标”的抽象层级并不一致。

## Goals / Non-Goals

**Goals:**
- 将 COA 控制台迁移为 `pages/` 下的独立子页面，并可从主站首页卡片进入。
- 让该页面在视觉、导航和文案结构上与现有作品集子页面保持一致。
- 将 GitHub Token 与 Gist ID 统一为本地连接配置项，并使用浏览器本地缓存回填。
- 保留原有控制台核心行为，包括状态读取、固定启动/停止动作、账号别名缓存、单人囚笼刷币开关和频率限制。

**Non-Goals:**
- 不修改 Gist 文件协议本身，不改变 `command.json` 与 `status.json` 的字段语义。
- 不引入构建工具、前端框架或后端代理层。
- 不在本次变更中扩展新的控制动作或重做交互流程。
- 不要求彻底移除旧根目录文件的所有兼容痕迹，但新的推荐入口必须是主站子页面。

## Decisions

### 1. 采用标准子页面目录承载 COA 控制台

页面将迁移到 `pages/coa-controller/index.html`，并使用与现有子页面一致的外层骨架：共享样式、返回首页入口、项目编号和简介区。

原因：
- 与当前站点的目录式路由约定一致，部署到静态托管后可直接通过 `/pages/coa-controller/` 访问。
- 让页面被主站首页索引，而不是继续作为根目录散落文件存在。
- 便于后续继续扩展更多工具页时复用同一套模板。

备选方案：
- 继续保留根目录 `COAController.html` 作为主实现。否决原因是会绕开已建立的页面体系。
- 仅在首页新增一个跳转到 `COAController.html` 的卡片。否决原因是入口虽然统一，但页面结构和资源组织仍然割裂。

### 2. 拆分页面结构与页面专属逻辑

`index.html` 只保留 COA 控制台页面骨架和交互所需的 DOM 节点，原内联脚本迁移到 `assets/js/pages/coa-controller.js`。页面继续通过原生浏览器 API 调用 GitHub Gist API，不引入额外依赖。

原因：
- 与现有“页面结构在 `pages/`，页面逻辑在 `assets/js/pages/`”的约定保持一致。
- 便于后续维护、调试和复用，也减少 HTML 文件中长段脚本的阅读成本。
- 让首页与该页面的资源边界更清晰。

备选方案：
- 继续使用单文件 HTML 内联脚本。短期改动更少，但不符合当前项目的资源组织约定。

### 3. 将 Gist 目标与认证信息统一为本地连接配置

新增 `GIST_ID` 输入框，并引入与 Token 类似的本地缓存键。页面在发起读取或写入前同时校验 Token 与 Gist ID；两者都只保存在当前浏览器本地，不写回 Gist。

原因：
- 使“连接到哪个 Gist”与“以什么身份访问”处于同一个配置层次，降低使用心智负担。
- 避免把环境配置写死在源码里，方便多环境、多 Gist 复用。
- 与当前 Token 本地缓存机制保持一致，减少额外学习成本。

备选方案：
- 继续写死 `GIST_ID`。否决原因是与 Token 的输入式配置模型不一致，也不利于不同使用环境切换。
- 允许通过 URL 参数传入 `GIST_ID`。可行但容易造成分享链接中泄漏目标配置，且不如显式输入稳定。

### 4. 在公共样式上补充控制台组件，而不是保留旧视觉

页面会复用 `site.css` 中已有的面板、标题、说明文字和按钮语言，并补充少量面向控制台页的表单网格、状态卡和操作区样式。颜色体系继续保持黑白灰主导，仅用克制的状态色表示成功、警告和错误。

原因：
- 用户已明确要求与主站风格一致，现有站点定位也是极简作品集风。
- 控制台仍需要表达运行态、警告和错误，少量语义色比完全黑白更利于识别，但不会破坏整体克制风格。
- 在公共样式中补充基础组件，后续类似工具页也可复用。

备选方案：
- 直接沿用原页面内联样式。否决原因是与站点现有视觉语言差异明显。
- 为该页单独创建完整私有样式文件。可行，但当前规模较小，共享组件沉淀到公共样式更合适。

### 5. 首页卡片仅声明页面入口，不承担页面业务配置

在 `assets/js/site-data.js` 中为 COA 控制台新增一条卡片数据，仅包含展示所需元信息。Token、Gist ID、账号别名等运行时配置全部留在页面本地缓存与页面脚本中处理。

原因：
- 首页只做入口聚合，符合既有架构约束。
- 避免把敏感或环境相关配置混入首页卡片元数据。

备选方案：
- 在卡片配置中预置 Gist ID。否决原因是配置依然会暴露在站点静态资源中，不符合本次目标。

## Risks / Trade-offs

- [根目录旧页面链接失效] → 可视情况保留一个轻量跳转页或提示页，但新入口统一迁移到子页面路径。
- [将脚本拆出后选择器或初始化顺序出错] → 保持 DOM id 基本不变，降低迁移风险，并在实现后做页面级诊断检查。
- [公共样式为控制台页增加组件后变得过重] → 只抽取通用布局和表单/状态卡样式，避免引入与单一业务强绑定的类名语义。
- [本地缓存新增 Gist ID 后用户可能遗留旧配置] → 提供清晰的配置说明，并允许单独清空 Token；如有必要可同时提供 Gist ID 的覆盖输入和即时回填。
- [静态前端直接访问 GitHub API 仍受 Token 权限限制] → 保持当前错误提示路径，明确将认证失败、缺失配置等问题反馈到页面状态区。

## Migration Plan

1. 新建 `pages/coa-controller/index.html`，使用当前站点子页面骨架组织 COA 控制台内容。
2. 新建 `assets/js/pages/coa-controller.js`，迁移原页面脚本并将 `GIST_ID` 改为输入式本地配置。
3. 在 `assets/css/site.css` 中补充该页面需要的表单、状态卡和操作区样式，并与现有视觉保持一致。
4. 在 `assets/js/site-data.js` 中新增 COA 控制台卡片，使其出现在首页项目列表中。
5. 视实现结果决定是否保留根目录 `COAController.html` 作为兼容跳转或提示入口，但不再承载主业务实现。
6. 验证首页卡片、子页面访问、连接配置缓存、状态刷新和动作提交逻辑。

回滚策略：
- 若新子页面实现异常，可临时恢复对根目录旧页面的直接使用。
- 因为页面仍是纯静态实现，回滚仅涉及文件内容恢复，不涉及数据迁移。
